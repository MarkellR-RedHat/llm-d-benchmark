Chapter One

The Test

We set up a real-world benchmark to answer one question: how much faster is cache-aware routing? The setup was straightforward -- three NVIDIA H200 GPUs, Meta's Llama 3.1 70B model, and two routing strategies going head-to-head across eight different workload patterns.

Measurement

In-pod measurement via direct HTTPS + streaming SSE. TTFT captured with time.perf_counter() from first byte of the SSE stream. Zero network overhead -- the benchmark client runs inside the cluster.

Routing Comparison

Round-robin: Standard Kubernetes Service load balancing -- requests distributed evenly across pods. Cache-aware: llm-d EPP routing -- requests routed to the pod with the best KV-cache match for that prompt's prefix.

Benchmark Suite

Eight tests covering context scaling (500-32K tokens), concurrent load (10 sessions), cache warmup curves, agentic chains (8 and 12 steps), multi-turn conversations, and fleet scaling projections.

Chapter Two

The Results

Eight benchmark tests measuring TTFT (time-to-first-token) across different real-world scenarios. Select any chart below to explore the data.

Context Length Scaling (500 - 32K tokens)

We started by measuring the fundamental relationship: how does response time change as prompts get longer? TTFT grows linearly with context length for cold starts (88ms to 3.7s), while cache hits stay nearly flat (33ms to 79ms). The widening gap is the value of cache-aware routing.

Key insight: At 32K tokens, cold-start TTFT is 3,742ms vs 79ms with a cache hit -- a 47.5x speedup. The advantage compounds as context windows grow.

Speedup Factor by Context Length

Taking those same numbers, we calculated the raw speedup at each context length. The advantage scales nearly linearly -- longer contexts mean bigger wins.

Key insight: Even at just 500 tokens, cache-aware routing is 2.7x faster. At 32K tokens, it reaches 47.5x -- and context windows keep growing.

Concurrent Load (10 sessions x 5 turns)

A single user is one thing. What happens when 10 users show up at once? Cache-aware routing maintains flat, predictable latency while round-robin oscillates wildly.

Key insight: Under production-like concurrency, cache-aware TTFT stays flat at 52-61ms while round-robin swings between 179-254ms. P95: 73ms vs 288ms.

Cache Warmup Curve (20 sequential requests)

How quickly does each routing strategy reach peak performance? Cache-aware converges immediately; round-robin needs one cold start per pod.

Key insight: Round-robin needs 3 cold starts (one per pod) before converging. With 100 pods in production, that becomes 100 cold starts. Cache-aware routing: always 1.

Extended Agentic Chain (12 steps, 7K-10K tokens)

AI agents chain 12 reasoning steps, each building on all prior context. Context grows with each step, making cache reuse increasingly valuable.

Key insight: Round-robin starts with a 679ms cold start, then stabilizes at ~150ms after pod warmup. Cache-aware stays at 65-73ms throughout -- 40% reduction at step 12.

Agentic Tool-Calling Chain (8 steps, 2.3K-4.4K tokens)

A realistic tool-calling workflow where each step adds to the conversation. TTFT diverges from the very first step as cache-aware routing reuses the system prompt cache.

Key insight: By step 4, round-robin is 3.1x slower (135ms vs 43ms). Each agent step builds on all prior context -- cache-aware routing avoids re-prefilling it.

Multi-Turn Conversations (10 turns, 3 conversations)

The most common pattern: back-and-forth conversation over 10 turns. Median TTFT across 3 independent conversations. Round-robin consistently runs 20-40% slower as it scatters turns across pods.

Key insight: With just 3 pods, the difference is moderate (20-40%). At production scale (50-100 pods), round-robin rarely hits the same pod twice, widening the gap dramatically.

Fleet Warmup Projection

What happens when you scale from 3 pods to 200? Round-robin warmup scales linearly with pod count; cache-aware stays constant regardless of fleet size.

Key insight: At 100 pods with 32K context: round-robin needs 374 seconds (6.2 minutes) of warmup. Cache-aware: 3.7 seconds. The gap is O(N) vs O(1).

Chapter Three

Why It Matters

No benchmark tells the whole story. Here is an honest look at where cache-aware routing delivers its biggest wins -- and where the advantage is smaller.

Where It Helps Most

  • Multi-turn conversations (each turn reuses all prior context)
  • Agentic chains (tool calls build on accumulated context)
  • Long-context workloads (RAG, document Q&A)
  • Large fleets (100+ pods amplify the cold-start problem)

Where It Helps Less

  • Shared prefixes across few pods (all pods cache it quickly)
  • Very short contexts under 500 tokens (2.7x vs 47.5x at 32K)
  • Single-shot classification with unique inputs (no context reuse)
  • The EPP adds ~1-2ms routing overhead per request
Note: When 20 users queried the same document via round-robin across 3 pods, all pods cached the shared prefix quickly. Sticky routing then concentrated load on 1 pod with no cache advantage, resulting in 15.7% worse performance. Cache-aware routing benefits per-session context, not globally shared prefixes at small scale.
Chapter Four

How It Works

The mechanism behind the numbers. Four steps explain why cache-aware routing delivers such dramatic improvements in time-to-first-token.

1

The Prefill Problem

Every LLM request starts with prefill -- processing the entire input context before generating the first output token. Prefill time scales linearly: 88ms at 500 tokens, 3.7 seconds at 32K tokens.

2

Prefix Caching

vLLM stores computed KV blocks in GPU memory. A cache hit skips prefill entirely, producing constant ~35-44ms TTFT regardless of context length -- but only if the request reaches the right pod.

3

Round-Robin Scatters

Standard Kubernetes load balancing distributes requests randomly. Each pod must re-compute prefill for prompts it hasn't seen. With 100 pods, 99% of requests are cache misses.

4

llm-d Routes Smart

llm-d's Endpoint Picker (EPP) routes each request to the pod with the best KV-cache match, turning every request into a cache hit. Warmup is O(1) instead of O(N pods).

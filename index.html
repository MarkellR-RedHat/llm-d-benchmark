Chapter Two -- The Results

Context Length Scaling

TTFT grows linearly with context length for cold starts -- 88ms at 500 tokens to 3.7 seconds at 32K tokens. Cache hits stay nearly flat: 33ms to 79ms. The widening gap is the value of cache-aware routing.

Key insight: At 32K tokens, cold-start TTFT is 3,742ms vs 79ms with a cache hit -- a 47.5x speedup. The advantage compounds as context windows grow.

How It Works
1

The Prefill Problem

Every LLM request starts with prefill -- processing the entire input before generating the first token. 88ms at 500 tokens, 3.7s at 32K.

2

Prefix Caching

vLLM stores computed KV blocks. A cache hit skips prefill entirely -- constant ~35-44ms TTFT regardless of context. But only if the request hits the right pod.

3

Round-Robin Scatters

Standard K8s load balancing distributes randomly. Each pod re-computes prefill for unseen prompts. With 100 pods, 99% are cache misses.

4

llm-d Routes Smart

llm-d's Endpoint Picker routes each request to the pod with the best KV-cache match. Every request becomes a cache hit. Warmup is O(1).

Where It Helps Most

  • Multi-turn conversations (each turn reuses all prior context)
  • Agentic chains (tool calls build on accumulated context)
  • Long-context workloads (RAG, document Q&A)
  • Large fleets (100+ pods amplify the cold-start problem)

Where It Helps Less

  • Shared prefixes across few pods (all pods cache it quickly)
  • Very short contexts under 500 tokens (2.7x vs 47.5x at 32K)
  • Single-shot classification with unique inputs (no reuse)
  • The EPP adds ~1-2ms routing overhead per request

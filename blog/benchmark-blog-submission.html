47x faster time-to-first-token: benchmarking llm-d's cache-aware routing on NVIDIA H200 GPUs

By Markell Rawls, Developer Advocate, Red Hat

If you're serving LLMs on Kubernetes with round-robin load balancing, every follow-up request in a conversation is recomputing context that was already computed on another pod. This is the prefill problem. Prefill—the phase where the model processes all prior tokens before generating a single new one—scales linearly with context length and dominates time-to-first-token (TTFT). Round-robin load balancing wastes this compute by scattering sequential requests across pods, forcing each pod to re-derive the KV cache from scratch. llm-d solves this with KV-cache-aware routing through its Endpoint Picker (EPP), which sends requests to the pod that already has the relevant KV cache in memory.

For setup instructions, see the companion blog Deploy llm-d in 5 minutes. For conceptual foundation, see What is llm-d?

What we tested

We designed a benchmark suite to isolate the impact of cache-aware routing under realistic inference conditions. Every measurement was taken inside the pod to eliminate network variables, giving us a clean signal on what the GPU is actually doing.

Model
Meta Llama 3.1 70B Instruct with FP8-dynamic quantization, published by Red Hat AI on Hugging Face
Hardware
3x NVIDIA H200 141GB HBM3e GPUs on a single Red Hat OpenShift node
Inference engine
vLLM with automatic prefix caching enabled
Measurement
In-pod measurement via direct HTTPS with streaming SSE parsing. time.perf_counter() for TTFT timing. Zero network overhead—no load balancer, no proxy, no route
Routing comparison
Round-robin (strict pod rotation simulating Kubernetes Service behavior) vs cache-aware (all session requests routed to one pod, simulating llm-d's EPP behavior)
Test suite
8 tests covering context scaling, concurrent load, agentic chains, multi-turn conversations, and cache warmup dynamics

The scaling advantage

The core result is straightforward: cache-aware routing gets faster relative to round-robin as context length increases. The table below shows TTFT for cold requests (cache miss, simulating round-robin) versus warm requests (cache hit, simulating cache-aware routing) across context lengths from 500 to 32,000 tokens.

Context (tokens) Cold TTFT Warm TTFT Speedup Savings
500 88ms 33ms 2.7x 63%
1,000 110ms 33ms 3.3x 70%
2,000 219ms 36ms 6.1x 84%
4,000 369ms 38ms 9.8x 90%
8,000 758ms 44ms 17.2x 94%
12,000 1,270ms 49ms 25.9x 96%
16,000 1,637ms 55ms 29.6x 97%
24,000 2,692ms 66ms 40.9x 98%
32,000 3,742ms 79ms 47.5x 98%

Cold TTFT scales linearly at approximately 0.095ms per token. Warm TTFT barely increases—from 33ms to 79ms across the entire range. The growing gap between these two lines is the value of cache-aware routing, and it grows with every token of context.

At 32,000 tokens—common for RAG with long documents—a cold-start request takes 3.7 seconds before the model generates a single token. With cache-aware routing, the same request produces its first token in 79 milliseconds.

As models move to 128K context windows, this advantage only grows.

Under production load

Synthetic single-request benchmarks are useful, but production inference servers handle many sessions simultaneously. We ran 10 concurrent sessions, each executing 5 conversational turns at the same time, to see how cache-aware routing performs under realistic contention.

Round-robin delivered a mean TTFT of 220ms with a P95 of 288ms. Cache-aware routing cut that to a mean of 55ms and a P95 of 73ms—a 75% reduction in TTFT. The advantage was larger under concurrent load than in isolated tests, not smaller. This makes sense: cache misses waste GPU compute on redundant prefill, and that redundant work competes with other in-flight requests for GPU cycles. Cache hits free the GPU to focus on decoding, which benefits everyone in the queue.

The per-turn breakdown tells an equally important story. Sticky routing stayed flat at 52–61ms across all 5 turns. Round-robin oscillated between 179–289ms depending on whether a given turn happened to land on a pod that had seen this session before. Predictable latency matters for user experience—users notice variance as much as they notice absolute delay.

Agentic workflows compound the benefit

We simulated a 12-step agentic chain processing a 7,000-token document. Context grew from 7,000 tokens at step 1 to 10,300 tokens by step 12 as tool calls and reasoning steps accumulated. This is a common pattern in production: retrieval-augmented generation followed by iterative analysis, summarization, and action planning.

Under round-robin routing, steps 1 through 3 averaged roughly 700ms each. This is the cold-start penalty: with 3 pods in the pool, each of the first 3 steps landed on a pod seeing the document for the first time. Steps 4 through 12 dropped to approximately 145ms as the document prefix became cached across all pods—but each pod still had to re-prefill the growing conversation context that followed the shared prefix.

Cache-aware routing stayed at 44–87ms throughout the entire chain. Because every step landed on the same pod, each step only prefilled the new tokens added since the last step—roughly 250 tokens per step, regardless of total context length. The result was a 40% TTFT reduction even in the late stages of the chain, where round-robin had already warmed its prefix caches.

In an agentic workflow, every tool call, every reasoning step, every follow-up question carries the ENTIRE prior conversation as context. Without cache-aware routing, each step re-prefills everything the previous step already computed—on a different pod.

What this means at fleet scale

Our benchmark used 3 pods, which is enough to demonstrate the pattern but small enough that round-robin warmup completes quickly. In production fleets with tens or hundreds of pods, the warmup cost becomes the dominant performance factor for new sessions and topics.

With 3 pods, round-robin needed 3 sequential cold starts—one per pod—before the fleet converged to cached performance for a given prefix. With cache-aware routing, convergence happened at request 1. The table below projects this effect to larger fleet sizes.

Pods RR Warmup (8K context) RR Warmup (32K context) Cache-aware
3 2.3s 11.2s 0.8s / 3.7s
10 7.6s 37.4s same
50 37.9s 187.1s same
100 75.8s 374.2s same

At 100 pods with a 32K-token prompt, round-robin needs over six minutes of sequential cold starts before the fleet is warm. Cache-aware routing: 3.7 seconds. One cold start, once, on one pod.

Where it doesn't help (and that's okay)

When 20 users queried the same shared document simultaneously, sticky routing was 15.7% slower than round-robin with 3 pods. The reason is straightforward: all 3 pods quickly cached the shared prefix through natural request distribution, so sticky routing just concentrated load on one pod unnecessarily while the other two sat partially idle.

Cache-aware routing's value is in per-session context—conversations, agent chains, user-specific RAG retrieval. Not in globally shared prefixes that all pods cache quickly. This is actually the correct behavior: the EPP should route to the best cache match, and when every pod has the same cache, it should distribute load evenly.

It is worth noting that with larger fleets (50+ pods), even shared prefixes benefit from cache-aware routing because fleet-wide warmup takes proportionally longer. The 15.7% penalty we observed is specific to small fleets where broadcast warmup is cheap.

In production, expect even more

Several factors would make the advantage more dramatic in production than what we measured on our 3-pod testbed. Fleets of 50 to 100 pods push warmup cost linearly higher, making cache-aware routing's constant-time warmup enormously valuable. Context windows of 32K to 128K tokens extend our speedup curve further—it showed no sign of plateauing at 47.5x and keeps climbing. The actual EPP is smarter than our simulation: it picks the pod with the best cache match across all active sessions, not just "the same pod." And production traffic combines agentic chains, multi-turn conversations, and RAG retrieval simultaneously, compounding the benefits we measured in isolation.

Some factors might temper the results. The EPP adds approximately 1–2ms of routing latency for its cache-matching logic. Short contexts under 500 tokens see a smaller absolute savings (2.7x rather than 47x). And single-shot completions with no context reuse—one-off summarization requests, for example—see minimal benefit from cache-aware routing because there is no cache to reuse.

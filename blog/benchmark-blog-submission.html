47x faster time-to-first-token: benchmarking llm-d's cache-aware routing on NVIDIA H200 GPUs

By Markell Rawls, Developer Advocate, Red Hat

If you're serving large language models on Kubernetes today, your inference stack probably has a problem you haven't noticed yet. Every follow-up message in a multi-turn conversation, every tool call in an agentic chain, every iterative question against a long document—all of them are recomputing context that was already computed seconds ago on a different pod.

This is the prefill problem. Before an LLM generates a single token, it has to process every prior token in the conversation. This prefill phase scales linearly with context length and dominates time-to-first-token (TTFT). Standard Kubernetes round-robin load balancing makes it worse by scattering sequential requests across pods, forcing each one to re-derive the KV cache from scratch.

llm-d solves this with KV-cache-aware routing. Its Endpoint Picker (EPP) sends each request to the pod that already has the relevant KV cache in GPU memory. Instead of recomputing, it reuses.

We benchmarked this on real hardware to find out how much it actually matters. The answer: a lot more than we expected.

New to llm-d? Read What is llm-d and why do we need it? for the conceptual foundation. For deployment instructions, see Deploy llm-d on Kubernetes in 5 minutes.

What we tested

We designed a benchmark suite to isolate the impact of cache-aware routing under realistic inference conditions. Every measurement was taken inside the pod—no load balancer, no proxy, no route—so the numbers reflect what the GPU is actually doing, not what the network adds.

Test suite

The scaling advantage

The core result is straightforward: cache-aware routing gets faster relative to round-robin as context length increases. Cold-start TTFT scales linearly at roughly 0.095ms per token. Cache-hit TTFT barely moves.

The growing gap between those two columns is the entire value proposition of cache-aware routing, and it grows with every token of context.

At 32,000 tokens—common for RAG with long documents—a cold-start request waits 3.7 seconds before the model generates a single token. With cache-aware routing, the same request produces its first token in 79 milliseconds.

As models move to 128K context windows, this advantage only grows. The curve showed no sign of plateauing at 47.5x.

Under production load

Single-request benchmarks are useful, but production inference servers handle many sessions at once. We ran 10 concurrent sessions, each executing 5 conversational turns simultaneously.

Round-robin delivered a mean TTFT of 220ms with a P95 of 288ms. Cache-aware routing: mean of 55ms, P95 of 73ms. That's a 75% reduction.

The advantage was larger under concurrent load, not smaller. This makes sense: cache misses waste GPU compute on redundant prefill, and that wasted work competes with other in-flight requests for GPU cycles. Cache hits free the GPU to focus on decoding, which benefits everyone in the queue.

The per-turn breakdown tells an equally important story. Cache-aware routing stayed flat at 52–61ms across all 5 turns. Round-robin oscillated between 179–289ms depending on whether a given turn happened to land on a pod that had already seen this session. Users notice variance as much as they notice absolute delay.

Agentic workflows compound the benefit

We simulated a 12-step agentic chain processing a 7,000-token document. Context grew from 7,000 tokens at step 1 to 10,300 tokens by step 12 as tool calls and reasoning steps accumulated. This is a common production pattern: RAG retrieval followed by iterative analysis, summarization, and action planning.

Under round-robin, the first 3 steps averaged roughly 700ms each—the cold-start penalty, with each step landing on a pod seeing the document for the first time. Steps 4 through 12 dropped to about 145ms as the prefix cached across all pods, but each pod still re-prefilled the growing conversation context beyond the shared prefix.

Cache-aware routing stayed at 44–87ms throughout the entire chain. Every step landed on the same pod, so each step only prefilled the ~250 new tokens added since the last step. The result: a 40% TTFT reduction even in late stages where round-robin had already warmed its prefix caches.

In an agentic workflow, every tool call carries the ENTIRE prior conversation as context. Without cache-aware routing, each step re-prefills everything the previous step already computed—on a different pod.

What this means at fleet scale

Our benchmark used 3 pods. That's enough to demonstrate the pattern but small enough that round-robin warmup completes quickly. In production fleets with tens or hundreds of pods, warmup cost becomes the dominant performance factor.

With 3 pods, round-robin needed 3 sequential cold starts before the fleet converged to cached performance. With cache-aware routing, convergence happened at request 1.

At 100 pods with a 32K-token prompt, round-robin needs over six minutes of sequential cold starts before the fleet is warm. Cache-aware routing: 3.7 seconds. One cold start, once, on one pod. The gap is O(N) vs O(1).

Where it doesn't help (and that's okay)

When 20 users queried the same shared document simultaneously, sticky routing was 15.7% slower than round-robin with 3 pods. All 3 pods quickly cached the shared prefix through natural request distribution, so sticky routing just concentrated load on one pod unnecessarily.

This is actually the correct behavior. Cache-aware routing's value is in per-session context—conversations, agent chains, user-specific RAG retrieval. For globally shared prefixes that all pods cache quickly, the EPP should distribute load evenly. And at larger fleet sizes (50+ pods), even shared prefixes benefit from cache-aware routing because fleet-wide warmup takes proportionally longer.

In production, expect even more

Several factors would make the advantage more dramatic in production than what we measured on our 3-pod testbed:

  • Fleet size: 50–100+ pods push warmup cost linearly higher, making cache-aware routing's constant-time warmup enormously valuable
  • Context windows: 32K–128K tokens extend the speedup curve further—it showed no sign of plateauing
  • Real EPP intelligence: The actual EPP picks the pod with the best cache match across all sessions, not just "the same pod" like our simulation
  • Mixed workloads: Production traffic combines agentic chains, multi-turn, and RAG simultaneously, compounding benefits we measured in isolation

Some factors temper the results. The EPP adds approximately 1–2ms of routing latency. Short contexts under 500 tokens see a smaller absolute savings (2.7x rather than 47x). And single-shot completions with no context reuse see minimal benefit.
